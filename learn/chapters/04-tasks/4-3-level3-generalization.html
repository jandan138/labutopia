Level3 generalization：object/material/instruction 的变化

Level3 让任务不再只围绕一个固定对象或材质。它通过 object list、position range、material_paths、inference test material 等机制制造泛化压力。

先讲人话

Level3 让任务不再只围绕一个固定对象或材质。它通过 object list、position range、material_paths、inference test material 等机制制造泛化压力。 换成更直白的话：这一节是在告诉你“遇到 Level3 generalization：object/material/instruction 的变化 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成学做实验动作：先练拿起、放下、倾倒，再把它们串成完整实验。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 tasks/base_task.py。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

BaseTask.setup_objects 会收集 cfg.task.obj_paths，update_object_and_material_indices 在成功 episode 后轮换 object/material。setup_materials 还区分 collect/infer 和 is_test_material，让测试时可以使用不同 material set。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

这对 policy training 很关键。模型如果只在一个颜色/材质/背景下训练，视觉 encoder 很容易学到捷径；Level3 的意义是让 data distribution 更接近 benchmark 对泛化的要求。

读论文和跑实验时怎么用

从 USD 资产角度，material path 和 object path 一样都是 contract。材质不兼容、绑定失败或 test_materials 缺失，都会让所谓 generalization 实验失真。

本节要记住
  • Generalization 不只是换 prompt。
  • Material binding 使用 UsdShade，需要 target/material prim 都有效。
  • Train/test material split 要在 config 中显式表达。
