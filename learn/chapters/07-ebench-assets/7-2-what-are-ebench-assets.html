EBench assets 是什么：外部 Hugging Face 资产包

EBench entry repo 本身并不把 benchmark assets 全部放在 Git 里；它把前门、baselines、scripts 留在 GitHub，把核心 assets 和 dataset 放到 Hugging Face。

先讲人话

EBench entry repo 本身并不把 benchmark assets 全部放在 Git 里；它把前门、baselines、scripts 留在 GitHub，把核心 assets 和 dataset 放到 Hugging Face。 换成更直白的话：这一节是在告诉你“遇到 EBench assets 是什么：外部 Hugging Face 资产包 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成搬家：东西可以搬，但门锁、插座、电压和房间布局不同，不能原封不动就开工。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 /cpfs/shared/simulation/zhuzihou/dev/EBench/README.md。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

EBench README 与 docs 指向 InternRobotics/EBench-Assets 和 InternRobotics/EBench-Dataset。这说明 EBench assets 是 benchmark runtime 的外部依赖，而不是像 LabUtopia 本地 assets 那样随 repo 直接出现。EBench 的 server 端来自 GenManip，client 侧用 genmanip-client 和 gmp 命令提交/评测。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

资产内容服务的是 EBench 的 26 tasks、3 tracks/splits、本体 lift2 和 LeRobot 数据格式。它和 baselines 的 openpi/X-VLA/InternVLA-A1 通过统一 observation/action contract 对接。

读论文和跑实验时怎么用

所以比较两边 assets 时，不能只比较文件扩展名。要比较“谁管理资产、谁加载资产、谁定义 task、谁记录数据、谁评分”。

本节要记住
  • EBench assets 外置在 Hugging Face。
  • Entry repo 不是完整 runtime。
  • Assets 与 GenManip server、gmp eval、LeRobot dataset 是一套系统。
