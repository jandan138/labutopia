Embodiment mismatch：Franka/ridgeback_franka 对 lift2

最硬的差异是 robot embodiment。LabUtopia 的主线是 Franka 与 ridgeback_franka；EBench 的评测 contract 是 lift2 dual-arm mobile base。

先讲人话

最硬的差异是 robot embodiment。LabUtopia 的主线是 Franka 与 ridgeback_franka；EBench 的评测 contract 是 lift2 dual-arm mobile base。 换成更直白的话：这一节是在告诉你“遇到 Embodiment mismatch：Franka/ridgeback_franka 对 lift2 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成搬家：东西可以搬，但门锁、插座、电压和房间布局不同，不能原封不动就开工。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 assets/robots/Franka.usd。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

Embodiment mismatch 不只是外观不同。它改变 joint count、gripper layout、base motion、camera placement、reachable workspace、action vector 和 success predicate。EBench docs 中 observation 包含 state.joints、state.gripper、state.base、state.ee_pose 与多视角 video；action 则包含双臂关节/夹爪和 base_motion。LabUtopia 的 Franka controller 和 RMPFlow contract 不等价。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

这意味着 LabUtopia robot assets 不能直接拿去跑 EBench baseline。更现实的方案是保留 EBench 的 lift2，把 LabUtopia 的 lab props/scene 作为新环境素材；或者在 GenManip 中新建一个支持 Franka/ridgeback_franka 的 benchmark family，但那已不是直接复用。

读论文和跑实验时怎么用

对于资产迁移，先问 robot 是否保持 EBench lift2。如果保持，LabUtopia 的桌面高度、object scale、collision、reachability 都要为 lift2 重新验证。

本节要记住
  • embodiment mismatch 是直接复用的第一阻碍。
  • Robot asset 影响 action/observation schema。
  • 建议先迁移 props/scene，而不是迁移 robot。
