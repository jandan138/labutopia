LabScene：scientific laboratory 的空间语义

LabScene 不是“漂亮的实验室背景”，而是把化学实验任务需要的 workspace、器材、台面、门抽屉、加热设备、导航障碍放进可引用的 USD hierarchy。

先讲人话

LabScene 不是“漂亮的实验室背景”，而是把化学实验任务需要的 workspace、器材、台面、门抽屉、加热设备、导航障碍放进可引用的 USD hierarchy。 换成更直白的话：这一节是在告诉你“遇到 LabScene：scientific laboratory 的空间语义 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成给实验室贴标签：同一间房既可以从论文看，也可以从代码看，标签帮你不迷路。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 assets/chemistry_lab。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

本地 assets 体现了这个定位：assets/chemistry_lab/lab_001/lab_001.usd、lab_003.usd、hard_task/Scene1_hard.usd 和 navigation_lab/navigation_lab_01/lab.usd 分别服务 table-top manipulation、press/heat/stir、long-horizon hard task 与 navigation/mobile manipulation。

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

场景语义通过 prim path 进入配置。比如 task config 中的 obj_paths、target_path、camera prim 和 material path，都是 LabScene 的接口。你在 Isaac Sim 里点一个物体，只是看到了 prim；在 LabUtopia 中，这个 prim 还可能决定 success predicate、data record 的 object_name 和 controller 的目标点。

读论文和跑实验时怎么用

这也解释了为什么资产迁移不能只复制 USD。只要目标平台依赖不同 prim naming、robot reachability、collision schema 或 task scoring，LabScene 的语义就需要重新绑定。

本节要记住
  • LabScene 的核心是 semantic scene，而不是纯 geometry。
  • USD path 是任务配置和运行时之间的桥。
  • 资产迁移时 prim path 稳定性比贴图是否漂亮更关键。
