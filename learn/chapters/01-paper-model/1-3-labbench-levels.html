LabBench：层级任务为什么要分 level

LabBench 的价值在于把科学实验能力拆成层级：atomic skill、combined skill、generalization、long horizon、mobility，不同 level 让 failure diagnosis 更具体。

先讲人话

LabBench 的价值在于把科学实验能力拆成层级：atomic skill、combined skill、generalization、long horizon、mobility，不同 level 让 failure diagnosis 更具体。 换成更直白的话：这一节是在告诉你“遇到 LabBench：层级任务为什么要分 level 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成给实验室贴标签：同一间房既可以从论文看，也可以从代码看，标签帮你不迷路。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 config/level1_pick.yaml。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

本地 config 目录把这种层级变成可运行入口。level1_pick.yaml、level1_pour.yaml、level1_stir.yaml 代表 atomic task；level2_PourLiquid.yaml 与 level2_TransportBeaker.yaml 开始组合技能；level4_CleanBeaker.yaml、level4_OpenTransportPour.yaml 则把任务拉成长序列；level5_Navigation.yaml 和 level5_Mobile_manipulation.yaml 引入移动底盘。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

从 RL 看，level 不是难度标签那么简单。它改变 horizon、state distribution、failure mode 和 demonstration 数据的覆盖需求。一个 policy 在 level1 pick 上成功，不代表它能在 level4 clean beaker 中忍受中间步骤的微小误差。

读论文和跑实验时怎么用

因此后续讲 controller 时，我们会把 level 当作 task decomposition 的线索，而不是文件名前缀。

本节要记住
  • Level 越高，episode horizon 和 state distribution 越复杂。
  • Generalization task 通常通过 object/material/background/instruction variation 制造分布外条件。
  • Mobile level 改变 robot embodiment 和 action space。
