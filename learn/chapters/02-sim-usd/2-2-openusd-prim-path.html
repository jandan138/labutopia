OpenUSD：prim path 为什么像 API

在 LabUtopia 里，USD 的 prim path 不只是 UI 中的树节点名字，它实际承担了对象寻址、camera 初始化、material binding 和 success check 的 API 角色。

先讲人话

在 LabUtopia 里，USD 的 prim path 不只是 UI 中的树节点名字，它实际承担了对象寻址、camera 初始化、material binding 和 success check 的 API 角色。 换成更直白的话：这一节是在告诉你“遇到 OpenUSD：prim path 为什么像 API 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成一场舞台剧：Stage 是舞台，World 是每一拍的节奏，SimulationApp 是剧场电源。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 tasks/base_task.py。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

OpenUSD 的核心是 stage/layer/prim/composition。你可以把 stage 想成场景数据库，把 prim path 想成数据库 key。LabUtopia 的 task config 把 object path、target path、camera prim path 写进 YAML，运行时用 stage.GetPrimAtPath 查询它们是否存在，然后把位置、尺寸、语义或材质绑定到这些 prim 上。

读论文和跑实验时怎么用

资产迁移到 EBench 时也会遇到同样问题。EBench/GenManip 的 task config 和 scoring 期待另一套 scene semantics；LabUtopia 的 prim path 不能默认被对方理解。

本节要记住
  • Prim path 是 contract，不是标签。
  • Reference 到 /World 后，路径会影响所有 lookup。
  • 迁移资产前先做 path inventory。
