为什么 LabUtopia 不是一个普通仿真 demo

如果你只把 LabUtopia 当成一批 USD 场景，就会错过它真正的结构：它同时是 scientific embodied agents 的仿真环境、任务层级和数据生成入口。

为什么它在本项目里重要

这份教程把项目当成一本书来拆。第一层是 Isaac Sim 与 OpenUSD 的世界观；第二层是 LabUtopia 论文里的 LabSim、LabScene、LabBench；第三层才是代码里的 BaseTask、BaseController、policy workspace 和 assets。读者如果有一点 RL 与 USD 资产基础，最容易卡住的不是概念，而是这些层之间的映射。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

本地代码显示，项目入口不是某个孤立 task，而是 main.py 中的 loop：启动 SimulationApp，创建 World，把配置里的 usd_path reference 到 /World，再让 task 提供 state、controller 产生 action。也就是说，LabUtopia 的“实验室”不是静态模型，而是一套从 USD 场景到 robot policy 的闭环。

读论文和跑实验时怎么用

读这本教程时，先不要急着训练 policy。你要先能回答三个问题：场景从哪里进来，state 如何被整理，action 由谁执行。能回答这三个问题，再去看 Diffusion Policy 或 Action Chunking Transformer，就不会把 learning algorithm 和 simulation plumbing 混在一起。

本节要记住
  • 把 LabUtopia 看成 benchmark/runtime/data 三者叠在一起，而不是单一 repo。
  • 中文解释负责建直觉；English terms 保留论文和代码真实命名。
  • 本书所有章节都尽量把论文术语落到本地文件路径。
