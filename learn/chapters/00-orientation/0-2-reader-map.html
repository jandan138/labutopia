先讲人话

这本书假设你知道 reward、policy、trajectory 这些 RL 词，也知道 Isaac Sim 里可以点开 USD stage，但还没有把 benchmark、dataset、controller 串起来。 换成更直白的话：这一节是在告诉你“遇到 读者画像：从 RL 和 USD 点选经验出发 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成先画地图：你不是马上拧螺丝，而是先知道门、桌子、机器人、数据出口分别在哪里。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 config/*.yaml。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

RL 背景给你一个很有用的类比：每个 episode 都像一次 rollout，task 负责提供 observation 和 terminal condition，controller 或 learned policy 负责 action。区别是 LabUtopia 目前大量 collect mode 由 rule/controller 驱动，它更像 imitation learning pipeline 的数据工厂，而不是端到端 RL trainer。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

USD 点选经验也有用。你知道 prim path、material、camera、reference 的存在，就能理解为什么配置里大量出现 /World/...、usd_path 和 camera prim。LabUtopia 的难点在于这些路径不是装饰，它们直接影响 object lookup、success check、data collection 和可复用性。

config/*.yaml
assets/chemistry_lab
assets/robots
policy/config

读论文和跑实验时怎么用

所以本教程会反复把同一件事讲三遍：先用直觉讲一遍，再用本地代码路径讲一遍，最后用“如果迁移到 EBench 会怎样”讲一遍。这样你不只是会运行命令，还能判断改动会撞到哪一层 contract。
