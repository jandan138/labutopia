新手跑起来：环境、踩坑和 expert 视频

这一节不讲大理论，只回答一个很实际的问题：一个第一次接手 LabUtopia 的人，怎样确认环境没跑偏、怎样启动一个任务、怎样只录 expert 轨迹视频，以及失败时先看哪里。

先讲人话

“跑起来”先不要理解成训练模型，也不要理解成完整生产 dataset。最稳的第一步是让 Isaac Sim 启动，跑一个短 expert episode，并看到任务成功或视频输出。

你可以这样想

Conda env 是工具箱，Isaac Sim 是大型实验室机器，Hydra config 是实验说明书。你要先确认拿的是同一个工具箱，再按同一份说明书开机器。

读完你应该能回答

  • 应该用哪个 Python 跑 LabUtopia？
  • 为什么每次要先 unset OMNI_SERVER？
  • 怎么区分“有视频”和“expert 成功”？

本机已经能跑 LabUtopia expert 轨迹，并完成了全量视频批跑。准确口径如下：

项目结果新手怎么理解

可读视频29/29每个任务都至少录到了一个 mp4。

第一步：确认自己站在正确目录，用正确 Python

LabUtopia 这类项目最常见的低级错误是：代码目录对了，但 Python 环境错了；或者 Python 环境对了，但跑到了另一个项目的 Isaac。先做这三个检查。

确认目录、GPU、Python
pwd
nvidia-smi
/cpfs/user/zhuzihou/conda-managed/envs/labutopia-py311/bin/python -V

当前已经验证过的 Python 是：

/cpfs/user/zhuzihou/conda-managed/envs/labutopia-py311/bin/python
Python 3.11.15
Isaac Sim 5.1 pip installation
Torch 2.9.0+cu126

当前 driver 是 570.153.02。它已经能完成短流程和本次视频批跑，但低于 Isaac Sim 5.1 文档中本次调研看到的 Linux tested driver 580.65.06。所以它是“本机能用”，不是“长期基线完全没风险”。

第二步：每次启动前先清掉旧 Isaac 影响

Isaac Sim 会读很多环境变量和 Omniverse cache。我们之前遇到过旧 OMNI_SERVER 指向旧内容，导致当前项目可能被旧 Isaac 资产污染。新手最稳的做法是每次运行前固定执行这三行。

每次运行前的固定三行
unset OMNI_SERVER
export OMNI_KIT_ACCEPT_EULA=YES
export PYTHONUNBUFFERED=1

这三行分别在干什么

  • unset OMNI_SERVER：不要再连旧 Nucleus 或旧 Isaac 内容。
  • OMNI_KIT_ACCEPT_EULA=YES：headless 运行时不要卡在许可确认。
  • PYTHONUNBUFFERED=1：日志实时输出，方便判断是卡住还是正在加载。

第三步：先跑最小 smoke，不要一上来全量跑

第一次验证环境时，不建议直接跑 29 个任务。先跑一个短任务，看 Isaac 能否启动、config 能否加载、controller 能否完成、程序能否正常退出。

最小 smoke
unset OMNI_SERVER
export OMNI_KIT_ACCEPT_EULA=YES
export PYTHONUNBUFFERED=1

/cpfs/user/zhuzihou/conda-managed/envs/labutopia-py311/bin/python \
  main.py \
  --config-dir docs/tmp/labutopia_smoke \
  --config-name level1_pick_smoke \
  --headless \
  --no-video

你希望看到类似结果：

成功信号
Success Rate = 1/1 (100.00%)

如果 smoke 都跑不起来，先不要调 task 逻辑。先回到环境层：Python 是否正确、driver 是否可见、Isaac package 是否在这个 env 里、旧 OMNI_SERVER 是否清掉。

第四步：只录 expert 视频，不写大 H5

正式数据采集会写 HDF5，慢且占空间。我们这次讨论的目标只是“跑 expert 轨迹并录视频”，所以使用 mock_collector。它的意思很简单：让 episode 正常推进，但不要真的写大 dataset。

单任务录视频模板
unset OMNI_SERVER
export OMNI_KIT_ACCEPT_EULA=YES
export PYTHONUNBUFFERED=1

/cpfs/user/zhuzihou/conda-managed/envs/labutopia-py311/bin/python \
  main.py \
  --config-dir docs/tmp/expert_video_configs/20260609_002039 \
  --config-name level1_pick \
  --headless \
  --video-dir artifacts/videos/manual/level1_pick

这里每个参数的含义：

参数作用为什么新手要关心
--config-dir告诉 Hydra 去哪里找临时 config。视频 config 和正式 config 要分开。
--config-name选择要跑的任务。比如 level1_pick、level5_Navigation。
--headless不打开 GUI 窗口。服务器上通常必须这样跑。
--video-dir指定 mp4 输出位置。否则视频会散在 Hydra output 目录里。

第五步：看结果时只看三件事

新手不要被 Isaac 日志里大量 warning 吓住。第一轮只判断三件事：

1. 进程是否正常退出

退出码是 0，通常说明这次任务没有 crash。

2. 是否看到 success

日志里出现成功率或 summary 里 success_seen=True，才算 expert 成功。

3. mp4 是否可读

视频能打开，只说明有画面；它不能替代 success 判断。

检查视频是否可读
ffprobe artifacts/videos/expert_trajectories/20260609_002039_final/level1_pick/episode_0.mp4

新手最容易踩的坑

症状最可能原因先做什么
一启动就找不到 Isaac/omni 模块Python env 用错了。显式使用 labutopia-py311/bin/python。
加载到旧资产或旧服务器OMNI_SERVER 或 cache 污染。运行前 unset OMNI_SERVER。
headless 卡住或退出很早EULA 或显示相关设置不对。设置 OMNI_KIT_ACCEPT_EULA=YES，使用 --headless。
视频窗口相关错误服务器无显示，但代码尝试 show video。headless 下只保存视频，不调用 GUI 显示。
utils import 很奇怪Isaac/cv2 路径里的顶层 utils 抢占了项目包。确认 project root 在 sys.path 前面，且项目有 utils/__init__.py。
日志里有 MDL/texture warning材质或贴图缺失。先判断任务是否成功；如果做 showcase，再修 asset。
有视频但 summary 不是 ok任务中途失败或 controller bug。按 video_only 记录，不要算成功。

下一步怎样变成真正规范流程

现在的状态是“本机能跑起来，并且有完整视频证据”。如果要让新同学或新机器稳定复现，还要补四件事。

环境锁定

把 Python、Isaac Sim、Torch、CUDA wheel、driver 建成 lock 或容器说明。

批跑脚本

新增 scripts/record_expert_videos.py，不要靠手工复制命令。

视频验证脚本

新增 scripts/validate_expert_videos.py，自动生成 summary 和 QA 状态。

本节要记住
  • 先 smoke，再单任务视频，最后才全量批跑。
  • 每次 Isaac 运行前先清 OMNI_SERVER，并显式设置 EULA。
  • 视频存在不等于 expert 成功，summary 状态才是最终口径。
  • 视频和 H5 是本机 artifact，不是应该提交的源码。
