训练 workspace：Hydra 到 BaseWorkspace

训练入口 train.py 很短，但它背后依赖 Hydra 根据配置实例化 workspace，再由 workspace 组织 dataset、policy、optimizer、checkpoint 和 validation。

先讲人话

训练入口 train.py 很短，但它背后依赖 Hydra 根据配置实例化 workspace，再由 workspace 组织 dataset、policy、optimizer、checkpoint 和 validation。 换成更直白的话：这一节是在告诉你“遇到 训练 workspace：Hydra 到 BaseWorkspace 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成给学生录示范视频：数据越清楚，后面训练出来的 policy 越不容易学偏。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 train.py。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

train.py 读取 policy/config 下的 YAML，解析 cfg._target_，用 hydra.utils.get_class 找到 workspace class，并调用 workspace.run()。这让 ACT 和 Diffusion Policy 可以共享训练入口，但使用不同 workspace。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

对初学者来说，训练报错通常不是 train.py 本身，而是 cfg 中的 dataset_path、shape_meta、policy target、obs key 与真实 HDF5 数据不一致。先用小 batch 打印 dataset sample，比盲目调模型更有效。

读论文和跑实验时怎么用

把训练和 simulation 分开理解也很重要：main.py 负责生成/运行 episode，train.py 负责离线训练。infer mode 才把训练好的模型带回 simulation。

本节要记住
  • train.py 是 dispatcher，不是训练算法主体。
  • cfg._target_ 决定实际 workspace。
  • Dataset path 和 shape_meta 是最常见错配点。
