Robots：Franka, ridgeback_franka, Fetch

LabUtopia 当前任务主要围绕 Franka 和 ridgeback_franka，Fetch 资产也存在，但本地 config 的主线是 Franka 桌面操作与 ridgebase 移动任务。

先讲人话

LabUtopia 当前任务主要围绕 Franka 和 ridgeback_franka，Fetch 资产也存在，但本地 config 的主线是 Franka 桌面操作与 ridgebase 移动任务。 换成更直白的话：这一节是在告诉你“遇到 Robots：Franka, ridgeback_franka, Fetch 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成实验室仓库：不是所有瓶子都一样，位置、材质、碰撞和名字都会影响能不能用。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 assets/robots/Franka.usd。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

assets/robots/Franka.usd 与 SubUSDs/panda_* 构成 Franka 机械臂资产；assets/robots/ridgeback_franka.usd 与 schema/subUSD 负责移动底盘加机械臂；assets/fetch 包含 URDF、USD 和 fixed camera variants。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

Robot asset 不只是模型。它决定 joint naming、articulation root、gripper geometry、RMPFlow descriptor、action dimension 和 controller 能否工作。LabUtopia 的 BaseController 默认创建 FrankaRMPFlowController；Level5 则切换到 ridgebase controller。

读论文和跑实验时怎么用

这也是和 EBench 最大差异之一：EBench 的本体是 lift2 dual-arm + mobile base，观测和动作 contract 都围绕它设计。LabUtopia 的 robot assets 不能直接替换 EBench 本体。

本节要记住
  • Robot asset 决定 control contract。
  • Franka 桌面任务和 ridgebase 移动任务要分开看。
  • Fetch 资产存在，不等于所有 runtime 都已围绕 Fetch 配好。
