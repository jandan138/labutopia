Collect mode 与 infer mode

同一个 controller base 同时支持 collect 和 infer，这让 LabUtopia 既能生成 demonstration，也能把训练好的 policy 放回 simulation 中测试。

先讲人话

同一个 controller base 同时支持 collect 和 infer，这让 LabUtopia 既能生成 demonstration，也能把训练好的 policy 放回 simulation 中测试。 换成更直白的话：这一节是在告诉你“遇到 Collect mode 与 infer mode 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成一次做实验的流水线：先拿实验单，再摆场景，再观察，再动作，再记录结果。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 controllers/base_controller.py。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

在 collect mode 下，BaseController 创建 DataCollector，controller 通常按规则或运动规划生成动作，并把 trajectory 写入 HDF5。 在 infer mode 下，BaseController 创建 FrankaTrajectoryController 和 InferenceEngine，由 local/remote/replay engine 提供动作。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

这条分支的意义是把“专家数据采集”和“模型部署评测”放在同一 runtime 中。它比单独写一个 offline trainer 更有价值，因为你可以回到相同 scene、camera 和 robot contract 下看 policy 是否真的能执行。

读论文和跑实验时怎么用

从调试角度，mode 错了会导致完全不同的错误。collect mode 没有模型推理，infer mode 则需要 checkpoint、shape_meta、trajectory controller 与 inference engine 全部匹配。

本节要记住
  • collect 生成数据，infer 消耗模型。
  • local/remote/replay 是 inference engine 的三种入口。
  • 同一 config 改 mode 不代表所有参数都自动适配。
