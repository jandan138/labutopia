Hydra config 是整条链的入口

LabUtopia 的 task 不是在命令行里硬编码出来的，而是由 Hydra config 决定 task_type、controller_type、usd_path、robot、camera 和 collect/infer mode。

为什么它在本项目里重要

main.py 用 hydra.initialize 和 hydra.compose 读取 --config-name。这份 cfg 随后被保存到 run_dir，成为一次实验的 provenance。你在 config 里改 usd_path，就换场景；改 robot.type，就换 factory 创建的机器人；改 mode，BaseController 初始化 collect 或 infer 分支。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

从工程角度看，Hydra config 是 LabUtopia 的 experiment contract。它把 Python class 的组合关系外置，让同一套 runtime 可以运行不同 level task。读项目时不要从 controller 文件海洋开始，而要先看 config，找到它指定的 task/controller/asset。

main.py
config/level1_pick.yaml
policy/config/train_diffusion_unet_image_workspace.yaml
Collect one task
python main.py --config-name level1_pick
python main.py --config-name level4_CleanBeaker --headless --no-video

读论文和跑实验时怎么用

这也解释了为什么 assets chapter 会频繁引用 config：资产不是孤立目录，而是 cfg.usd_path 和 prim paths 引入 runtime。

本节要记住
  • 先读 config，再读 factory，再读具体 task/controller。
  • run_dir 中保存 config.yaml，方便复现实验。
  • Hydra resolver 用于训练配置里的动态表达式。
