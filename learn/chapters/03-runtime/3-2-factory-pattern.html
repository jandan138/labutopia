Factory pattern：字符串如何变成 class

配置里写的是 task_type 和 controller_type，但运行时需要具体 Python class。LabUtopia 用 factory 把这一步集中管理。

先讲人话

配置里写的是 task_type 和 controller_type，但运行时需要具体 Python class。LabUtopia 用 factory 把这一步集中管理。 换成更直白的话：这一节是在告诉你“遇到 Factory pattern：字符串如何变成 class 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成一次做实验的流水线：先拿实验单，再摆场景，再观察，再动作，再记录结果。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 factories/task_factory.py。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

create_robot、create_task、create_controller 分别根据 cfg 的字符串创建对象。这样做的好处是 main loop 不需要知道 pick、pour、navigation、cleanbeaker 的细节；缺点是新任务如果没有注册到 factory，会在运行时才暴露。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

从阅读路线看，factory 是你查找具体实现的索引。看到 controller_type: "pickpour"，就去 controller factory 找映射，再读 controllers/pickpour_controller.py。看到 task_type: "navigation"，就去 task factory 找到 tasks/navigation_task.py。

读论文和跑实验时怎么用

这类模式在 benchmark 项目里很常见，因为任务数量多，但 runtime loop 应该稳定。你改 factory 就是在扩大支持的 task vocabulary。

本节要记住
  • Factory 让 main.py 不随 task 数量膨胀。
  • 新增 task 要同时补 config、task class、controller class、factory mapping。
  • 迁移到 EBench 时，GenManip 的 task registry 会成为对应位置。
