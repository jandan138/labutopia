先讲人话

LabUtopia 的 README 已经给出设计哲学：scene state 和 observation acquisition 由 tasks 处理，robot control 与 success condition checking 由 controllers 处理。 换成更直白的话：这一节是在告诉你“遇到 Task/Controller boundary：谁看世界，谁做动作 时，应该先看哪一层，不要一上来被术语吓住”。

你可以这样想

把它想成一次做实验的流水线：先拿实验单，再摆场景，再观察，再动作，再记录结果。

读完你应该能回答

  • 这节和一次 episode、一个 asset 或一条数据有什么关系？
  • 如果要在项目里找证据，第一眼应该打开 tasks/base_task.py。
  • 如果这里出错，后面的 task、controller、dataset 或 EBench 复用会受什么影响？

为什么它在本项目里重要

BaseTask 负责 cameras、objects、materials、state dict。它知道如何从 stage 中读取 object position、size、camera frame，也知道什么时候需要 reset。BaseController 负责把 state 变成 action，管理 collect/infer mode、RMPFlow、DataCollector、episode count 和 failure reason。

从 RL 看 把概念落到 trajectory

这里的关键不是把术语背下来，而是看它怎样改变 episode、observation、action、success 和 dataset distribution。

从代码和资产落地

这个边界对初学者很有帮助。你看到一个 bug，如果是 camera image 为空、object path 找不到、material 没绑定，优先看 task；如果是 robot 不动、动作顺序错、success 统计异常，优先看 controller。不要一上来全局搜索。

读论文和跑实验时怎么用

从 RL 看，task 更像 environment observation wrapper，controller 更像 expert policy 或 learned policy adapter。二者之间的接口就是 state dict。

本节要记住
  • Task owns sensing and scene bookkeeping。
  • Controller owns action generation and episode policy。
  • State dict 是二者之间的 contract。
