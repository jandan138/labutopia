1. 先把 LabUtopia 摆正

你可能已经会在 Isaac Sim 里打开场景、点点机器人、看 USD asset 的层级，也听过 reinforcement learning 里的 environment、observation、action、reward。但 LabUtopia 这个仓库如果直接按 Gym-style RL environment 去读，会很快迷路。它更像一个围绕 Isaac Sim 搭起来的 scientific embodied agents benchmark 与 imitation-learning data pipeline。

论文层面，LabUtopia 由三个名字组成：LabSim、LabScene、LabBench。项目页和 OpenReview 摘要都强调：它面向 scientific laboratory，比家庭场景更强调物理-化学变化、长序列 planning、仪器与容器之间的细粒度操作。你在本仓库看到的核心实现，是围绕这些实验室任务进行 simulation、data collection、training 和 inference。

这意味着我们读代码时不要先问“reward 在哪里”，而要先问四个更贴近本仓库的问题：

  • 一个 YAML config 如何选择 USD scene、robot、task、controller、camera、collector？
  • task.step 每一帧收集了什么 observation/state？
  • controller.step 如何把 state 转成 robot action，并判断 done / is_success？
  • successful episode 如何写成 HDF5 dataset，再喂给 ACT 或 Diffusion Policy？
一个重要区分 Isaac Lab 是 NVIDIA 官方的 robot-learning framework，强调大规模并行环境、RL/IL training workflows。LabUtopia 当前仓库则是直接在 Isaac Sim 应用与 Python API 上组织实验室 benchmark、demonstration collection、policy training 和 inference。两者都在 Isaac Sim 生态里，但抽象边界不同。

2. 论文里的三层心智模型

LabUtopia paper 把问题放在 scientific embodied agents 上：实验室任务不像“把杯子从桌上拿起”那么单一。一个 agent 可能要识别容器、移动到仪器旁、抓取烧杯、倾倒、搅拌、加热、等待状态变化、再继续下一步。这样的环境需要同时处理 perception、planning、control 和 domain-specific state change。

Component 01

LabSim

高保真 simulator。项目页称它基于 Isaac Sim，并扩展 chemical engine，用来表达颜色变化、产物生成等实验室现象。本仓库的 assets/chemistry_lab/、main.py、tasks/ 和 controllers/ 都服务于这层。

Component 02

LabScene

场景生成与资产组织。当前仓库里你看到的是已经存在的 USD scenes 与 assets，例如 assets/chemistry_lab/lab_001/lab_001.usd、assets/navigation_lab/navigation_lab_01/lab.usd。

Component 03

LabBench

分层 benchmark。README 与 config 文件体现了 Level 1 到 Level 5：atomic actions、short-horizon manipulation、generalization、long-horizon manipulation、mobile manipulation。

资料版本之间有一个小差异：OpenReview 与 arXiv 摘要写 LabUtopia supports 30 distinct tasks and 200+ assets；项目页在 LabBench 段落中写 over 50 tasks。教程里遇到论文级统计时，我以 OpenReview/arXiv 摘要为主，同时提醒你这是版本差异。实际读本地仓库时，我们以 config/*.yaml、factories/*_factory.py 注册表和具体 controller/task 文件为准。

3. 仓库地图：先分清谁负责什么

如果你只从文件名看，会觉得这个仓库很散：tasks/、controllers/、policy/、assets/、data_collectors/ 都在说机器人。但它其实有很稳定的分工。

读代码最有效的方式，是把它想象成一条流水线：config 不是“参数列表”，而是一张组装说明；factory 不是复杂设计模式，而是把字符串变成具体 class 的入口；task 和 controller 共同定义一个 episode；collector 决定成功 demonstration 如何落盘；policy 再从 dataset 里学习 action chunk。

4. Isaac Sim Runtime：为什么 main.py 要这样写

Isaac Sim 的 standalone Python workflow 有一个硬规则：先创建 SimulationApp，再 import 许多 omni / isaacsim 模块。本仓库的 main.py 正是这样开头。它先解析 CLI，然后用 config dict 创建 SimulationApp。这一步不是普通 Python 初始化，而是在启动 Omniverse Kit application。

main.py · runtime skeletonlocal code reference
from isaacsim import SimulationApp

simulation_config = {
    "headless": False,
    "extra_args": ["--/rtx/raytracing/fractionalCutoutOpacity=true"],
}

simulation_app = SimulationApp(simulation_config)

from isaacsim.core.api import World
from isaacsim.core.utils.stage import add_reference_to_stage

World 是 Isaac Sim Core API 里用来管理 physics step、reset、scene objects 的高层对象。LabUtopia 根据 CLI --backend 选择 CPU/numpy 或 GPU-related setting。随后它调用 add_reference_to_stage(usd_path=..., prim_path="/World") 把 YAML 指向的 USD scene 作为 reference 加到当前 Stage。

这里要把 OpenUSD 的术语和 Isaac Sim 的 runtime 连起来：Stage 是合成后的场景，Prim 是场景树节点，Layer 是可保存的场景描述文件。LabUtopia 不在 Python 里手写一个完整实验室，而是把 USD scene 引入 Stage，再用 task/controller 在这个 Stage 上移动物体、绑定材质、读取相机、控制 robot。

这个 stepper 对应本地 main.py 的真实顺序。你可以把它背下来：SimulationApp 先开门，Hydra 给说明书，World 管仿真时间，USD reference 放入实验室，factory 拼出 robot/task/controller，loop 每帧推进。

5. Config 是这套系统的装配图

LabUtopia 使用 YAML config 控制任务。以 level1_pick.yaml 为例，它不是只存超参数，而是决定整个实验：任务类型是 pick，controller 类型也是 pick，mode 是 collect，USD path 指向 assets/chemistry_lab/lab_001/lab_001.usd，camera 包含外部视角与手眼相机，collector 选择 default 且 compression 是 gzip。

交互 · config explorer tabs

你读每个 config 时，可以按这个顺序扫：

  1. task_type 和 controller_type：它们分别会被 factories/task_factory.py 与 factories/controller_factory.py 解析。
  2. mode：collect 会启用 DataCollector，infer 会启用 InferenceEngine。
  3. usd_path：这是实际加载到 Stage 的场景。
  4. task 段：object paths、position ranges、max steps、navigation config 等任务语义。
  5. cameras：observation 的视觉来源，最终进入 HDF5 和 policy dataset。
  6. robot：当前支持 franka 和 ridgebase registry。
  7. hydra.run.dir / multi_run.run_dir：输出路径，后面 dataset/video/config 都落在这里。

6. task 与 controller 的边界

这个项目最重要的工程边界，就是 task 与 controller。很多初学者会把它们都理解成“任务代码”，结果读到一半就分不清状态、动作、成功条件到底在哪里。

交互 · boundary switch task vs controller

tasks/base_task.py

负责 cameras、objects、materials、reset、state dict。它更像“环境观察员”：告诉 controller 现在看到了什么、机器人关节在哪里、物体中心在哪里、相机图像是什么。

  • setup_cameras() 创建 Isaac Sim camera sensor。
  • get_camera_data() 调用 process_camera_image()。
  • get_basic_state_info() 组装 camera/state/object 信息。
  • check_frame_limits() 控制 episode 最大步数。

controllers/base_controller.py

负责 action、done、success、data collection、inference。它更像“执行者”：读 state，决定下一步关节 action，判断是否完成，成功时写 dataset。

  • _init_collect_mode() 创建 collector。
  • _init_infer_mode() 创建 trajectory 与 inference engine。
  • step(state) 返回 (action, done, is_success)。
  • reset() 更新 episode count 与 success rate。

以 PickTaskController 为例，PickTask 本身几乎只继承 SingleObjectTask，状态收集由 base/single object 逻辑完成；真正的 pick motion 在 controllers/pick_controller.py 里。collect mode 下，它调用 atomic PickController.forward()，并把 camera images 和 joint angles 交给 self.data_collector.cache_step()。当物体 z 坐标比初始高度高出阈值并持续足够步数，就认为 success，然后调用 write_cached_data()。

7. Episode Loop：一帧里发生了什么

main.py 的 while loop 是整套系统的心跳。它每次先 world.step(render=True)，然后根据 World 状态和 reset flags 决定是否 reset。真正的业务逻辑只有几行：state = task.step()，action, done, is_success = task_controller.step(state)，最后如果 action 不为空，就 robot.get_articulation_controller().apply_action(action)。

Robotapply_action

这条链路有两个 reset 来源：task_controller.need_reset() 和 task.need_reset()。controller 可能因为动作序列完成、成功判定通过、推理超时或失败而请求 reset；task 可能因为 frame limit、物体/材质切换、任务完成而请求 reset。reset 时先释放 video writer，再 task_controller.reset()，检查 episode 数量是否达到 max_episodes，最后 task.reset()。

注意一个微妙点：video saving 和 dataset writing 不是同一件事。video 是 main.py 根据 state['camera_display'] 叠图写 MP4；dataset 是 controller 在成功轨迹上通过 DataCollector 写 HDF5。一个 episode 有视频不代表它一定写入训练数据，失败 demonstration 会被 clear cache。

8. Data Collection：HDF5 里到底装了什么

在 collect mode 下，BaseController._init_collect_mode() 会调用 create_collector(cfg.collector.type, ...)。默认 collector 是 data_collectors/data_collector.py 里的 DataCollector。它先在输出目录下创建 dataset/ 和 dataset/meta/，然后按 camera config 初始化临时缓存。

每一帧，controller 调用 cache_step(camera_images, joint_angles, language_instruction)。成功时，write_cached_data(final_joint_positions) 把缓存转成 numpy arrays，创建 episode_0000.h5，并在 meta/episode.jsonl 里记录 episode index、task instruction、length 和 task properties。写 HDF5 的过程用 ProcessPoolExecutor 异步提交，所以 close() 时必须等待 pending futures。

这里的 actions 很容易误解。默认 DataCollector 在 write_cached_data() 中把 temp_agent_pose[1:] + [final_joint_positions] 当作 action。这是一种 imitation learning 常见处理：给定当前 observation，目标 action 是下一步或未来 joint positions。对于 navigation 这种 state/action 不同步的场景，仓库还有 ActionStateDataCollector，它的 cache_step() 会同时接收 joint state 与 action。

为什么要有 language instruction 一些 controller 会动态生成自然语言指令。例如 PickTaskController.get_language_instruction() 会从 object name 生成 “Pick up the ... from the table”。这让 dataset 能服务 vision-language-action 或 language-conditioned policy。

9. Policy Training：ACT 与 Diffusion Policy

LabUtopia 的训练入口是 train.py。它使用 Hydra 的 @hydra.main(..., config_path='policy/config') 读取训练 config，然后根据 cfg._target_ 找到 workspace class，最后 workspace.run()。这里的 Hydra 是配置组合与 object instantiation 体系，不只是 YAML parser。

train.py · workspace dispatchlocal code reference
@hydra.main(version_base=None, config_path="policy/config")
def main(cfg):
    OmegaConf.resolve(cfg)
    cls = hydra.utils.get_class(cfg._target_)
    workspace = cls(cfg)
    workspace.run()

仓库里两个主要训练方向是 Diffusion Policy 和 Action Chunking Transformer。Diffusion Policy 把 action sequence 当作条件 denoising diffusion process 来生成，论文强调它适合 multimodal action distributions、high-dimensional action spaces，并配合 receding horizon control。ACT 来自 ALOHA 相关工作，核心思想是让 transformer 一次预测一段 action chunk，缓解逐步自回归控制中的误差积累。

ConfigPolicy classDataset class读法
policy/config/train_diffusion_unet_image_workspace.yamlDiffusionUnetImagePolicyDPImageDataset用多帧 observation conditioning，预测一段 action horizon。核心参数包括 horizon、n_obs_steps、n_action_steps、num_inference_steps。
policy/config/train_act_image_workspace.yamlACTImagePolicyACTImageDataset用 transformer + action chunk。核心参数包括 num_queries、hidden_dim、num_layers、robot_state_dim、action_dim。

dataset class 做了三件关键事情。第一，打开 HDF5 episode 文件；第二，把每个 episode 切成训练 sequence；第三，返回 obs 和 action tensors。DPImageDataset 会返回类似 {obs: {camera_1_rgb, camera_2_rgb, agent_pose}, action} 的结构。normalizer 则对 action、agent_pose 和 images 建立尺度转换，保证 policy 训练时数值稳定。

训练代码不是 Isaac Sim runtime train.py 不启动 SimulationApp。它读的是已经采集好的 HDF5 dataset。也就是说，simulation/data collection 与 policy training 是两个阶段：前者需要 Isaac Sim runtime，后者主要是 PyTorch/Hydra pipeline。

10. Inference：local、remote、replay 三种入口

当 config 里 mode: "infer" 时，BaseController 会走 _init_infer_mode()，创建 FrankaTrajectoryController 和 InferenceEngine。工厂入口在 controllers/inference_engines/inference_engine_factory.py，支持 local、remote、replay。

infer.type = local

LocalModelInferenceEngine

加载 PyTorch checkpoint、policy config、normalizer，调用 policy.predict_action(obs_dict)。适合在本机 GPU 上跑训练好的 ACT/Diffusion policy。

infer.type = remote

RemoteInferenceEngine

通过 openpi_client.websocket_client_policy.WebsocketClientPolicy 连接远端 server，把 observation 和 language instruction 发过去，拿回 action。

infer.type = replay

ReplayInferenceEngine

从 HDF5 里读取已采集 action，按顺序 replay。适合 debug simulation/controller/data alignment。

BaseInferenceEngine 的核心机制是 observation history。它根据 cfg.infer.obs_names 把 camera names 映射到 policy obs keys，用 deque 保存 n_obs_steps 帧。当 trajectory controller 完成上一段轨迹，并且历史 observation 足够时，它准备 tensor dict，调用 _predict_action()，再把预测出的 action chunk 交给 trajectory controller 分帧执行。

11. LabBench 五级任务：从 atomic 到 mobile manipulation

本仓库的 config/ 文件非常适合用来理解 LabBench。Level 不是抽象概念，而是任务复杂度逐层上升。

Level 1

Atomic Manipulation Tasks

单个原子动作，如 level1_pick.yaml、level1_place.yaml、level1_pour.yaml、level1_press.yaml、level1_stir.yaml。它们通常由一个 atomic controller 完成。

Level 2

Short-Horizon Manipulation Tasks

2-3 个 atomic actions 组合，如 transport beaker、pour liquid、stir glass rod。重点是动作衔接与物体状态保持。

Level 3

Generalizable Short Manipulation Tasks

仍是短任务，但加入 object/material/layout variation，评估 ID/OOD generalization。config 中常体现为多个 object paths、material paths 或测试材质设置。

Level 4

Long-Horizon Manipulation Tasks

长序列实验流程，如 level4_CleanBeaker.yaml、level4_DeviceOperation.yaml、level4_LiquidMixing.yaml。controller 内部常串联多个 atomic actions。

Level 5

Mobile Manipulation Tasks

把 navigation 和 manipulation 放在一起，如 level5_Navigation.yaml、level5_Mobile_manipulation.yaml。需要 mobile base、waypoints、occupancy map 或 path planning。

12. Case Study A：level1_pick

Level 1 pick 是最适合入门的任务。它的 config 指向 assets/chemistry_lab/lab_001/lab_001.usd，目标物体路径是 /World/conical_bottle02，位置范围在桌面上的一小块区域。三个 cameras 分别提供两个外部视角和一个 wrist camera。

运行 python main.py --config-name level1_pick 后，系统会创建 Franka，加载 lab scene，创建 PickTask 和 PickTaskController。每次 reset 时，目标物体位置会被随机化。每帧 task 返回 object position、object size、joint positions、gripper position 和 camera data。controller 读取这些状态后调用 atomic PickController.forward() 产生关节 action。

成功判定也很朴素：物体 z 坐标比初始高度高出 0.1，并持续 REQUIRED_SUCCESS_STEPS。这个设计对初学者很友好，因为它把 “pick succeeded” 转成了一个可以在 state 里直接检查的几何条件。

13. Case Study B：Level4_CleanBeaker

Level4_CleanBeaker 展示了 long-horizon manipulation 的典型写法。它不是一个 neural policy 一步端到端解决所有事，而是在 collect mode 下用 task-level controller 串联一串 atomic controllers 来生成 demonstration。

CleanBeakerTask.reset() 会随机化两个 beaker 和两个 platform 的位置，并绑定 table materials。CleanBeakerTask.step() 每帧返回 joint positions、target position、beaker positions、platform positions、camera data 和 gripper position。它提供“看见世界”的信息，但不决定下一步动作。

CleanBeakerTaskController 内部有 7 个阶段：pick beaker2、pour beaker2 to beaker1、place beaker2、pick beaker1、shake beaker1、pour beaker1 to target beaker、place beaker1。每个阶段由一个 atomic controller 负责；当当前 atomic controller is_done()，就推进 _current_step。最后通过两个 beaker 是否回到对应 platform 附近来判定 success。

StepAtomic Controller为什么重要
1PickController抓起第二个 beaker，建立后续倾倒条件。
2PourController把内容倒入第一个 beaker，模拟实验流程中的 transfer。
3PlaceController放回第二个 beaker，避免后续操作干扰。
4PickController抓起第一个 beaker。
5ShakeController对应混合或清洗动作。
6PourController倒入 target beaker。
7PlaceController放回第一个 beaker，并进入 success check。

这个 case 告诉你：LabUtopia 的 benchmark 复杂性并不只来自模型，而来自 environment state、object relation、动作顺序、成功条件和 observation logging 的组合。

14. Case Study C：level5_Navigation

Navigation 任务与 manipulation 的代码形态明显不同。NavigationTask 会读取 task.navigation_config_path，加载 navigation assets 和 obstacle grid，再用 A* planning 生成从 start 到 end 的 waypoints。它还会把每个 waypoint 补上朝向 theta，并把 robot 初始位置设置到 start point。

NavigationController 使用 RidgebaseController 跟踪 waypoints。collect mode 下，它会把 start/end positions、num_waypoints 写进 task properties，并在每帧缓存 camera images、robot pose 和 action。与默认 DataCollector 不同，这里更适合 ActionStateDataCollector 的思想：navigation action 与 state 需要更明确地同步记录。

如果你有 RL 背景，这里很容易想到 reward shaping、collision penalty、goal distance reward。但本仓库当前 navigation collection 更像 classical planner + low-level controller 生成 demonstration，再用于 policy learning 或 replay。这个差异非常关键。

15. OpenUSD 与 assets：场景不是普通图片背景

LabUtopia 依赖 USD，因为实验室不是一张 background image，而是一个可组合、可引用、可绑定物理属性和材质的 scene graph。OpenUSD 文档强调 Stage、Prim、Property、Layer 和 composition arcs。在 Isaac Sim 中，这些概念直接影响机器人能否看到、碰到、抓住、推动、倒出物体。

本仓库通过 add_reference_to_stage(usd_path=os.path.abspath(cfg.usd_path), prim_path="/World") 把 USD scene 接入当前 Stage。随后 ObjectUtils 通过 prim path 查询几何中心、object size、设置 object position。camera config 里的 prim_path 也同样是 USD path，例如 /World/Camera1 或 /World/Franka/panda_hand/arm_camera。

读 USD path 的方法 每个 path 都是 scene graph 的定位符。/World/conical_bottle02 不是字符串标签，而是 Stage 中一个 Prim 的路径。controller 的 success check、task 的 randomization、camera 的 semantic segmentation 都依赖这些 path 正确存在。

16. Debugging：出问题时从哪里下手

LabUtopia 的 bug 往往不是单点 bug，而是 config、USD path、camera shape、controller state、dataset shape 之间错了一环。下面是实用排查顺序。

  1. 先看 config 是否装配正确。确认 task_type 和 controller_type 在 factory registry 中存在，usd_path 文件存在，camera prim_path 与 scene 一致。
  2. 再看 Stage 是否加载。如果 object path 无效，ObjectUtils 读到的位置/size 可能为空，success check 会失败。
  3. 检查 camera output。process_camera_image() 输出的数据是否符合 dataset 期待的 shape。训练配置通常假设 [3, 256, 256] 或类似 image shape。
  4. 检查 controller reset。如果 episode 卡死，看 need_reset()、check_frame_limits()、atomic controller 的 is_done()。
  5. 检查 HDF5 内容。确认 camera_1_rgb、camera_2_rgb、agent_pose、actions 长度一致，且 episode.jsonl 记录 length。
  6. 检查 training config。dataset_path 是否指向包含 HDF5 episode 的 dataset/，shape_meta 是否和实际 camera/joint/action shape 对齐。

17. 如何新增一个任务

新增任务时，不要先复制一个巨大的 controller 然后边跑边改。更稳的方式是从接口边界开始设计。

  1. 写一个新的 YAML config，先指定 usd_path、robot、cameras、task_type、controller_type、collector。
  2. 创建 task class，继承 BaseTask 或现有 single/dual object task。先让 step() 返回足够清楚的 state dict。
  3. 创建 controller class，继承 BaseController。collect mode 下先用 atomic actions 生成可靠 demonstration。
  4. 在 factories/task_factory.py 和 factories/controller_factory.py 注册字符串。
  5. 跑少量 episodes，打开视频和 HDF5 检查 observation/action alignment。
  6. 确认数据稳定后再接入 policy/config/task/*.yaml 和训练 workspace。

真正的工程原则是：先让 symbolic/controller pipeline 稳定地产生 demonstration，再训练 policy。否则你看到的 policy failure 可能只是数据坏、相机坏、success check 坏，和模型能力无关。

18. Glossary：边读边查

交互 · glossary search filter
SimulationApp

Isaac Sim standalone Python workflow 的应用入口，必须先启动再使用许多 omni/isaacsim 模块。

World

Isaac Sim Core API 的高层仿真对象，负责 step、reset 和 scene 管理。

Stage

OpenUSD 合成后的场景。LabUtopia 把 USD scene reference 到 /World。

task

负责 reset、camera、object/material state 和每帧 observation/state dict。

controller

负责把 state 转为 action，并判断 done/success。collect mode 还负责写 dataset。

DataCollector

缓存 successful trajectory，并写入 HDF5 episode 与 episode.jsonl metadata。

Diffusion Policy

把 robot action sequence 表示为 conditional denoising diffusion process 的 visuomotor policy。

Action Chunking Transformer

一次预测一段 action chunk 的 imitation learning policy，常简称 ACT。

Hydra

配置组合与对象实例化工具。LabUtopia training 用 _target_ 找 workspace/policy/dataset class。

InferenceEngine

封装 local checkpoint、remote OpenPI server 或 replay HDF5 actions 的推理入口。
