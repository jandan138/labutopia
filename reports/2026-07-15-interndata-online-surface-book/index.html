当 PBD 粒子
需要看起来像水

这本小书只讲一件事：在 LabUtopia 的湿实验倒液链路里，Isaac PhysX 的离散 particles 如何在每一次 observation 被翻译成一张闭合、可渲染的液体 triangle mesh——以及为什么这张 mesh 绝不能反过来当任务裁判。

受众假设：你有一点 Games101（标量场、插值、隐式曲面）基础；不必先读完整本 LabUtopia Interactive Book。中文负责把故事讲顺，English 保留架构术语、API、指标与论文概念名。

01 · perception gap

为什么粒子看起来不像水

先从一个几乎人人都会犯的直觉开始。你在仿真器里看见一坨液体粒子在动，心里会说：「这就是水。」但相机和 VLA 模型看到的，往往是另一回事：一群带透明度的小球、一条断断续续的珠串、或者倒水瞬间变成「玻璃沙」。物理上它们可能完全正确；视觉上它们不像连续液体。

原因很朴素。PhysX 的 PBD（Position-Based Dynamics）粒子系统存的是离散状态：位置、速度、相位、约束。它回答的是「质量在哪里、下一步往哪」。它不承诺给你一张光滑、封闭、可光照的液面。把 3600 个点直接画出来，等于把模拟器的内部表示泄漏到 observation 里。

于是 LabUtopia 引入一层克制的翻译：从当前帧的 particles 估计一个连续 density field，再抽取等值面，得到 presentation mesh。这层翻译不改物理、不造假碰撞、也不参与 scoring。

Interactive · 同一份状态，两种表达

切换按钮。青色点是 raw PBD particles（打分证据）；紫色连续形是由粒子派生的 presentation surface（仅供观看）。

常见误解

「网格更像水，所以成功条件应该看网格。」错。网格已经过 splat、Gaussian filter、gap closing 与 iso_level——它是视觉派生物。若用它打分，你等于把展示参数写进了任务规则。LabUtopia 的合同写得很死：scoring uses raw particles only。

把这句话记住，后面所有细节都围绕它转：reconstruction 是化妆间，不是法庭。

02 · mental model

三层心智模型：Physics / Reconstruction / Observation

工程上最容易晕的地方，不是某个公式，而是「这一层到底有没有权力」。请先把系统拆成三层，再读代码。

1 · Physics

Isaac PhysX PBD particles、容器 collider、重力与接触。权威属性：physxParticle:simulationPoints。它回答：液体实际在哪、怎样运动。

2 · Reconstruction

reconstruct_surface_live：lattice → density → Marching Cubes → closed mesh。它回答：这一帧怎样呈现给眼睛。

3 · Observation

online_fluid_surface：USD author、render barrier、cameras。它回答：控制器/模型此刻读到的图像是否绑定当前帧。

注意最右侧那一格：score 从 particles 旁路出来，不经过 mesh。这不是疏忽，是刻意的证据隔离（evidence isolation）。

合同边界

Reconstruction 是 one-way projection：Particles → mesh。没有 mesh → physics 的反馈边。它也不把 InternDataEngine、LabBuilder 或 SimFoundry 当作 runtime dependency；公开 InternData 只提供 behavior / visual inspiration，落地合同属于 LabUtopia。

用一句话概括整本书的立场：

物理说真话，重建说人话，观察说给模型听——三者必须同帧，但绝不能抢麦克风。

03 · Games101 bridge

从标量场到等值面：你已经有一半直觉

Games101 里，隐式曲面常常写成 f(x) = 0。液面重建只是把这个想法搬到「由粒子估计出来的密度场」上：先得到空间标量场 ρ(x)，再取

ρ(x) = iso_level　　→　　这就是“水面”

二维时，你在网格上做的是 Marching Squares：看每个格子四角相对阈值的内/外，连出 contour。三维时，Lorensen & Cline（1987）的 Marching Cubes 把同样的 divide-and-conquer 推到八个角、三角形面片。它最初服务 CT/MRI 体数据可视化；粒子液体可视化只是同一工具的另一应用场景。

Games101 桥

你不需要先成为流体专家。只要记住三步：采样场 → 选阈值 → 抽取边界。LabUtopia 的额外工程，是把这三步做成 deterministic、可 hash、可与物理同帧绑定的 contract。

Interactive · 2D Marching Squares lab

合成 density field 被规则网格采样；紫色折线是 f(x)=iso_level。拖动阈值，看“水面”收缩/扩张——这正是三维 Marching Cubes 的二维亲戚。

LabUtopia live path 里，这个阈值写作 iso_level = 0.02（密度场归一化尺度下的小数）。它调外观厚薄，不改粒子坐标，因此也不改 transfer score。

为什么还要 Gaussian 与 gap closing？

粒子场很“刺”。直接 MC 往往得到麻点状、带小洞的表面。工程上通常先把质量泼到格子（splat），再平滑（Gaussian），必要时做形态学补缝（grey closing）。这些步骤都在密度场上操作；它们属于 presentation pipeline，不是新的物理力。

04 · inspiration ≠ dependency

InternData 说了什么，LabUtopia 接了什么

InternData-A1（arXiv:2511.16651，CVPR 2026）是大规模高保真合成操作数据工作。论文在资产库描述里明确写到：fluid objects 用 particle-based dynamics 建模；粒子可在容器内自适应生成；液面通过 isosurface rendering 重建，并配合多样 PBD materials。

“Fluid objects are modeled with particle-based dynamics, where particles are adaptively generated within containers, and liquid surfaces reconstructed via isosurface rendering with diverse PBD materials.” — InternData-A1

这句话给 LabUtopia 的启发很具体：倒液不仅要物理可读，还要视觉可读——连贯水体 + 可辨认水柱，而不是珠串。但启发不是依赖：

InternData（公开主张）

大规模数据引擎里的流体资产配方：粒子动力学 + isosurface 呈现 + PBD materials。目标是 generalist policy 预训练数据。

LabUtopia（本地合同）

自己的 deterministic reconstruction contract：固定 spacing、padding、guards、gap-closing、manifold checks、frame token；与 online evaluation loop 咬合。

本地证据也对齐了 InternData-style 的流体尺度：约 3 mm contact/spacing、约 3600 particles、接近 12×12×25 的初始化网格。这些数字帮助标定“看起来对”的观察表面，但它们写在 LabUtopia 的 run evidence / contract 里，不是把第三方引擎嵌进核心包层。

边界再强调

PhysX SDK 提供原生 PxSparseGridIsosurfaceExtractor / PxIsosurfaceParams（可配置 isolevel、grid filtering、Taubin smoothing 等）。LabUtopia 的 online model-facing path 刻意不把 Isaac Sim 4.1 native PhysX isosurface 当作唯一呈现权威；它把当前帧自己重建的 mesh author 进 USD，以便合同可测、可复现、可与 particle score 隔离。

05 · temporal contract

在线一帧：时间才是真正的合同

很多人以为 reconstruction 难在几何。其实 online 模式最难的是同帧因果：相机图必须来自“刚结束的那次物理状态”，不能拿缓存 mesh 糊弄，也不能在 render 时偷偷再推进一步物理。

LabUtopia 的 online physics-surface evaluation loop 把一帧钉成：

action t → 4 × PhysX substeps (1/120 s) → state t+1 readback → surface t+1 → render barrier → observation t+1

也就是说，一次 logical observation ≈ 1/30 s，内部由四个 integration substeps 组成。权威粒子只允许读 physxParticle:simulationPoints，形状严格 3600×3、全有限。

播放或拖动。物理在第 2 格结束；之后只是把同一真相交给视觉与相机。render barrier 必须证明 timeline/physics 没有被渲染推进。

  • Reset observation：episode 开头可以没有 prior action；之后每帧要能指出 caused_by_action_index。
  • Freshness：hold 时粒子 hash 可以不变；新鲜度来自 step counters 与 render token，不是强迫状态哈希变化。
  • Failure：呈现失败会隐藏旧 mesh、抑制模型观察——绝不展示过期液体。
06 · reconstruct_surface_live

重建流水线：六个小决定

入口在 tools/labutopia_fluid/interndata_surface_reconstruction.py 的 reconstruct_surface_live。把它想成一条单向流水线；每一步都有明确失败方式。

physxParticle:simulationPoints (finite 3600×3) → compute_lattice_spec(AABB + padding_voxels=16, voxel_spacing_m=0.00135) → guards: axis≤384, total≤10_000_000 → trilinear splat (unit mass) → Gaussian filter σ≈1.60 voxels → live grey_closing size [7,7,17] (gravity-Z; no support expansion) → Marching Cubes (skimage lewiner, iso_level=0.02, gradient_direction=ascent) → closed manifold + outward normals + density 不触 lattice boundary → geometry_sha256 → frame token

1) Lattice：给液体画鞋盒，再切成格子

compute_lattice_spec 取粒子世界坐标的 min/max，按 voxel_spacing_m 量化，再向两侧扩展 padding_voxels。体素数是 Nx × Ny × Nz——跟粒子个数无关，跟粒子散开的空间跨度有关。杯子里一团水：鞋盒小；空中一条长水柱：鞋盒细长，总数爆炸。

2) Splat：把单位质量温柔分给邻居

三维 trilinear splat 把每个粒子的单位质量分到周围 8 个 lattice vertices。下面用 2D bilinear analog（4 个角）帮你看见权重：移动粒子，四个角的贡献会变，但总和始终为 1。

Interactive · Trilinear splat（2D analog）

三维只是再乘一次 w 方向权重：8 = 2×2×2。LabUtopia 用 np.add.at 累加，保证确定性。

3) Smooth & close：让水柱可读，但不许“画大”

Gaussian（σ≈1.60 voxels）抑制麻点。live contract 接着用矩形 grey closing [7,7,17]（Z 方向更长，对齐重力）弥合液柱内部空隙。合同极严：若 closing 扩张了 density 的 support bounds，直接报错。也就是说——可以补洞，不能偷偷把液体画到粒子 AABB 之外。

4) Marching Cubes + 卫生检查

在 iso_level=0.02 抽三角网格后，还要确认：非空、封闭流形、法向朝外、密度未顶到 lattice 边界。通过后计算 geometry_sha256，作为 surface frame token 的一部分，供 online runtime 证明「相机看到的就是这一帧刚 author 的表面」。

07 · capacity guard

1000 万体素：我们定的护栏，不是 Isaac 的天条

先回答你最关心的那句：这个上限是 LabUtopia live reconstruction contract 自己写的 max_total_voxels = 10_000_000，外加 max_axis_voxels = 384。超出就抛 lattice_total_limit_exceeded / lattice_axis_limit_exceeded。它不是 Isaac Sim 内核硬编码，也不是 PhysX isosurface API 的强制配额；它是在线密度场的内存/时延保险丝。

离线回放合同更紧：axis 256、total 8,000,000。live 放宽一点，是因为倒液跨度更大、又要赶 observation 节拍。

产品含义

倒液越“好看”（水柱越长越连贯），重建预算越吃紧。这不是物理失败，而是 observation capacity 问题。修复应抬高/重审 live budget，或缩小不必要的 AABB，而不是偷偷改轨迹让水柱变短来“过关”。

08 · two evidence lanes

打分车道与呈现车道

把粒子按空间关系分桶：source / target / transit / tabletop_spill / below_table……这是 scoring。把同一批粒子变成 mesh 给人看——这是 presentation。两条车道共享输入，不共享裁决权。

09 · incidents

两则真实风格的失败故事

故事 A ·「倒得很好，但示范不落盘」

液体 transfer 指标漂亮，但机械臂控制器自己的 return / home 条件没过。系统拒绝把这条轨迹当专家示范写入 H5。根因在 controller completion contract，不在液面重建。

故事 B ·「84% 还在空中就被打分？」

第一眼像 premature scoring：controller 结束时大量粒子仍在 transit。更精确的定位是：长水柱阶段，在线液面重建的 lattice 下一帧会超过 10M voxels，进程以 lattice_total_limit_exceeded 退出。看起来像“打分时机很怪”，实质是 presentation pipeline 先崩了。

修复方向（合同友好）

1）成功判定与倒液目标对齐（例如同一套 +Y 5 cm pour target）；2）把 live max_total_voxels 提高到足以覆盖该水柱；3）不改物理、不改倒液轨迹、不加飞溅限制来“躲”预算。先补失败测试，再改实现。

排查口诀：先分清是 controller、particle score 还是 surface budget。三件事症状可以长得很像，合同却完全不同。

11 · glossary

Glossary

PBD
Position-Based Dynamics：以位置约束迭代的粒子物理方法（Müller et al.）。
simulationPoints
PhysX 粒子系统权威位置属性 physxParticle:simulationPoints。
lattice
包住当前粒子 AABB 的规则 voxel grid；live 有 axis/total 护栏。
trilinear splat
把粒子质量按三线性权重写入 8 个邻居格子。
iso_level
等值面阈值；调外观，不判成功。
Marching Cubes
从 3D 标量场抽取三角等值面的经典算法（Lorensen & Cline 1987）。
grey_closing
形态学闭运算；live 用于补液柱缝隙，禁止扩张 support。
frame token
绑定 episode/step/粒子 hash/几何 hash 的帧身份，证明同帧呈现。
render barrier
渲染前后证明物理与 timeline 未推进的同步栅栏。
max_total_voxels
LabUtopia live 自定总预算，默认 10,000,000。
