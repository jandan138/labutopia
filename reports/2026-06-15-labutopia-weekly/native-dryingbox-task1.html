为什么 LabUtopia 原生能打开，但 EBench wrapper 可能失败？

因为这两个入口组合 USD 的方式不同。LabUtopia 原生场景通常是整份 lab_001.usd 一起打开，/World/DryingBox_01、/World/Looks、MDL 文件、贴图和灯光都在同一个大场景语境里。EBench wrapper 往往只把任务需要的对象包进自己的 runtime 路径；如果只引用 /World/DryingBox_01，mesh 的材质绑定还可能指向外面的 /World/Looks/...，这时画面可能退回默认色或 fallback。

LabUtopia 原生打开 像打开完整实验室工程：资产、材质库、贴图相对路径和场景上下文都在，很多跨 subtree 的引用天然可见。
EBench wrapper 打开 像把其中一个设备搬进标准评测箱：wrapper 必须显式拥有引用、材质、贴图、joint、camera 和 metric contract。
不能只看 render 默认蓝色或 displayColor 能让物体被看见，但不等于 native MDL/texture 成功；必须看 runtime readback 字段。

当前 material 选择：这次没有让 wrapper 隐式去找全局 /World/Looks。做法是 preserve_owned_world_looks：把源场景 /World/Looks payload 到 DryingBox wrapper 内部的 Looks scope，再把 task-visible mesh 的 material:binding 改到 wrapper 内部路径。Stage 5 又在 runtime 里读取了 ComputeBoundMaterial、MDL source/subIdentifier、resolved MDL path 和 hash；但 Aluminum remote waiver 仍让 full material closure 保持 open。

material:binding mesh 指向材质的地址。它可以写在 mesh 本身，也可能继承自父级 Xform、collection 或 GeomSubset。
ComputeBoundMaterial USD/Isaac 实际计算“这个 mesh 最后用哪个材质”。比只看一条关系更可靠。
empty binding 有 material:binding 这条关系，但 targets 是空的。它是材质缺口，不等于“指向了一个不存在的材质”。

这次 Stage 5 material 复核的新结论：wrapper 里 32 条 task-visible material record 已经进入 runtime readback，没有 unresolved binding target；但 Aluminum_Anodized_Charcoal.mdl 仍是 explicit waiver，native_material_closure_claim_allowed=false。因此蓝色门和画面可读性可以作为任务证据，不能写成 full native MDL/texture material closure。

Task 1 查到的主要风险

下面这些不是“页面不好看”的问题，而是 EBench/Isaac runtime 可能无法稳定 reset、step、score 或正确加载 native material 的问题。每一项都要在后续 acceptance stage 里被关闭或显式隔离。

Root scale 不是 1：/World/DryingBox_01 根节点有 xformOp:scale=[0.001, 0.001, 0.001]。这说明原生资产内部可能以毫米或其他单位建模，再通过根节点缩放到米制世界。

这不一定是错误，但在 EBench wrapper 中必须明确 unit policy。否则门轴、把手位置、碰撞体和机器人 reachability 都可能因为尺度解释不同而出问题。

有一个 fixed joint 指向了非 rigid body：/World/DryingBox_01/FixedJoint_01 的 physics:body0 指向 /World/DryingBox_01/Group_02/group/mesh，但 audit 发现该 target 没有 PhysicsRigidBodyAPI。

对产品来说，这相当于“图纸上有一颗螺丝连接到不存在的受力零件”。在静态 render 里可能看不出来，但进入 PhysX 后可能导致 warning、漂移或约束失效。

部分刚体质量和惯量无效：door mesh、body mesh、handle mesh 出现 zero_inertia、无效 centerOfMass 或零四元数 principalAxes；door 和 handle 的 mass 是 0。

这会影响 native-only smoke。我们不能只说“USD 能打开”，还要证明 120 个 physics steps 后 root pose、handle pose、joint positions 都是 finite。

不是只有门一个 active DOF：门是 PhysicsRevoluteJoint，axis 是 Y，limit 是 0..120；按钮还有 PhysicsPrismaticJoint，axis 也是 Y，limit 约 0..0.01。

这说明后续 metric 不能简单取“第一个 joint”或假设只有一个 DOF。EBench open_door 评分要明确绑定门的 RevoluteJoint，按钮 DOF 要隔离或显式忽略。

/World/Looks 正常，但 wrapper 里不能隐式依赖它：LabUtopia 原生场景里很多 mesh binding 指向 scene-level material library。整份场景打开时这没问题；但 EBench wrapper 如果只带走 /World/DryingBox_01，这些 binding 可能指向 wrapper 外部。

蓝色不能只凭肉眼定性：如果 runtime readback 证明 native material 已 resolved，蓝色可能就是原始材质；如果 readback 显示 displayColor 或 default fallback，蓝色只是兜底色。Stage 5 已在 EvalClient 渲染路径里确认画面、MDL source/subIdentifier、resolved MDL path/hash 和 metric evidence；但 Aluminum remote waiver 仍然存在，所以不能说 full native MDL/texture material closure。

这次要修的是统计口径：Stage 2 里两个 mesh 是显式空 material:binding targets，另一个 mesh 没有 authored binding。它们要分别记为 empty_authored_binding 和 unbound，不能混进真正的 unresolved_binding_target。

实习生小词典：读 USD articulation 时先搞懂这些词

prim：USD 里的一个节点

可以把 prim 理解成场景树里的一个对象节点。它可能只是分组 Xform，也可能是可见 Mesh，还可能带 physics schema。Task 1 先数清楚 54 个 prim，是为了确认我们面对的是原生复杂设备。

PhysicsArticulationRootAPI：告诉 PhysX 这是一组有关节的结构

如果没有 articulation root，门、按钮和刚体就很难被当作一个关节系统管理。Task 1 发现 root 在 /World/DryingBox_01，这是继续接入的好信号。

joint relationship：关节连接到哪两个刚体

physics:body0 和 physics:body1 必须指向存在且有 PhysicsRigidBodyAPI 的 prim。Task 1 发现 FixedJoint_01 有一个 target 缺少 rigid body，这是后续必须处理的明确风险。

DOF：评分时到底读哪个自由度

原生 DryingBox 同时有门的 RevoluteJoint 和按钮的 PrismaticJoint。因此 open_door metric 必须显式绑定门 joint，不能简单读取第一个 joint position。

状态词怎么理解？

attempted 已经跑过，有证据，但至少一个必要子项还没被接受。后续 Stage 7 如果发现 observation/action/camera/reward/logging 子项未闭环，会使用这个状态。
blocked 有前置条件、环境入口、资产包装或证据字段没闭环。它不是“项目失败”，只是下一步要解决的 blocker。
passed 该 stage 的必要字段都 PASS，且 JSON、log、截图或 readback 证据能复现。只有 passed 才能升级产品口径。
